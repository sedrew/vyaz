Для создания TypeScript-модели, которая описывает параграф с поддержкой богатого текста (Rich Text), автоподбора размера шрифта (Autofit font size) и переноса строк (Wrap), нам нужно спроектировать гибкую структуру данных.

Обычно Rich Text представляется в виде **параграфа (Paragraph)**, который содержит массив **текстовых фрагментов (Runs/Spans)**. Сам параграф отвечает за блочные свойства (выравнивание, autofit, wrap), а фрагменты — за инлайн-стили (жирный, курсив, цвет, конкретный размер шрифта).

Ниже представлена готовая TypeScript-модель с подробным описанием всех входных данных.

---

## TypeScript Модель

```typescript
// 1. Перечисление для типов выравнивания текста
export type TextAlignment = 'left' | 'center' | 'right' | 'justify';

// 2. Интерфейс для настройки Autofit (автоподбора размера)
export interface AutofitConfig {
  enabled: boolean;
  minFontSize?: number; // Минимальный порог, ниже которого шрифт не опустится
  maxFontSize?: number; // Максимальный базовый размер
}

// 3. Интерфейс для отдельного фрагмента стилизованного текста (Rich Text Run)
export interface TextRun {
  text: string;          // Сам текст фрагмента
  bold?: boolean;        // Жирный
  italic?: boolean;      // Курсив
  underline?: boolean;   // Подчеркнутый
  color?: string;        // Цвет в формате HEX, RGB или названия (#FFFFFF)
  fontSize?: number;     // Базовый размер шрифта для этого фрагмента (в px/pt)
  fontFamily?: string;   // Семейство шрифтов
}

// 4. Основной интерфейс для Параграфа
export interface RichTextParagraph {
  id: string;                    // Уникальный идентификатор параграфа
  children: TextRun[];          // Массив фрагментов текста (Rich Text)
  
  // Блочные свойства параграфа
  alignment: TextAlignment;     // Выравнивание (left, center, и т.д.)
  wrap: boolean;                // Разрешен ли перенос строк (true - wrap, false - no-wrap/clip)
  
  // Конфигурация автоподбора размера
  autofit: AutofitConfig;
  
  // Дополнительные полезные свойства для верстки (опционально)
  lineHeight?: number;          // Межстрочный интервал
  marginBottom?: number;        // Отступ после параграфа
}

```

---

## Описание входных данных (Входной JSON-объект)

Чтобы инициализировать или передать такой параграф в компонент, входной объект должен выглядеть следующим образом:

```json
{
  "id": "p-unique-123",
  "alignment": "center",
  "wrap": true,
  "autofit": {
    "enabled": true,
    "minFontSize": 12,
    "maxFontSize": 24
  },
  "children": [
    {
      "text": "Привет! Это "
    },
    {
      "text": "жирный и подчёркнутый",
      "bold": true,
      "underline": true,
      "color": "#FF0000"
    },
    {
      "text": " текст внутри одного параграфа.",
      "italic": true
    }
  ]
}

```

### Разбор параметров:

1. **`alignment` (Выравнивание):** Строка. Определяет, как текст прижимается к краям контейнера.
2. **`wrap` (Перенос строк):** Логическое (`true`/`false`). Если `true`, длинный текст переносится на новую строку при достижении границ контейнера. Если `false`, он либо обрезается (`clip`), либо уходит в `overflow`, либо сжимается за счет `autofit`.
3. **`autofit` (Автоподбор размера):** Объект.
* `enabled`: если `true`, то движок рендеринга должен динамически уменьшать `fontSize` всего параграфа, если текст не помещается в заданные границы (ширину/высоту контейнера).
* `minFontSize`: ограничитель снизу, чтобы текст не стал нечитаемым (микроскопическим).


4. **`children` (Содержимое Rich Text):** Массив объектов. Позволяет внутри одной строки комбинировать разные стили (например, сделать одно слово красным, а другое — жирным), что и является сутью Rich Text.

Абсолютно верный вопрос. Чтобы свойства **`wrap` (перенос)** и **`autofit` (автоподбор размера)** вообще могли работать, системе критически важно знать границы, в которые этот текст нужно вписать. Сам по себе параграф не знает, где он находится.

В архитектуре Rich Text приложений эта задача решается разделением на **модель данных** (содержимое) и **контекст контейнера / холста** (геометрия).

Есть два основных подхода, как передать эти размеры.

---

## Вариант 1. Размеры задаются на уровне контейнера параграфа (Рекомендуемый)

Размеры холста или текстового блока не принадлежат самому тексту — они принадлежат **контейнеру (рамке/текстовому боксу)**, в который этот текст налит.

В модель параграфа (или в модель текстового блока, который управляет параграфами) добавляются свойства `width` и `height`.

```typescript
export interface TextBoxModel {
  id: string;
  // Геометрия холста/контейнера, в котором рендерится текст
  width: number;       // Например, в пикселях (400px)
  height: number;      // Высота контейнера (опционально, для autofit по высоте)
  
  // Сам параграф (или массив параграфов)
  paragraph: RichTextParagraph; 
}

```

### Как это работает в логике кода:

1. **Если `wrap: true`:** Алгоритм рендеринга берет `width` контейнера (например, `300px`). Он начинает выстраивать слова фрагментов (`TextRun`) друг за другом. Как только суммарная ширина слов превышает `300px`, алгоритм переносит следующее слово на новую строку.
2. **Если `autofit.enabled: true`:** Алгоритм смотрит на `width` и `height`. Если весь текст с текущим размером шрифта (например, `24px`) не помещается в эти границы, алгоритм временно уменьшает шрифт до `23px`, проверяет снова, и так до тех пор, пока текст не влезет (или не упрется в `minFontSize`).

---

## Вариант 2. Размеры передаются извне в функцию расчета (Рендерер)

Если вы делаете графический редактор (вроде Canva, Figma или Photoshop), размеры холста постоянно меняются, когда пользователь тянет за край текстовой рамки. В этом случае размеры вообще не хранят в модели текста, их передают динамически в функцию, которая этот текст просчитывает (Layout Engine).

```typescript
interface LayoutOptions {
  maxWidth: number;   // Доступная ширина холста для этого параграфа
  maxHeight?: number; // Доступная высота холста
}

// Функция, которая рассчитывает перенос строк и autofit перед отрисовкой
function computeTextLayout(paragraph: RichTextParagraph, options: LayoutOptions) {
  const { maxWidth, maxHeight } = options;
  
  if (paragraph.wrap) {
    // Логика переноса: переносить строки, как только они достигают maxWidth
  }
  
  if (paragraph.autofit.enabled) {
    // Логика autofit: уменьшать fontSize, пока текст не впишется в maxWidth/maxHeight
  }
}

```

## Итог: Что нужно добавить?

Если вы проектируете полную модель для UI-компонента, вам нужно **объединить геометрию контейнера и стили текста**.

Вот как выглядит финальная входная модель для сущности «Текстовый блок на холсте»:

```typescript
export interface CanvasTextBlock {
  id: string;
  // 1. Координаты и размеры на холсте
  x: number;
  y: number;
  width: number;  // <--- ВОТ ОН, КЛЮЧ ДЛЯ WRAP И AUTOFIT
  height: number; // <--- Нужен для autofit, чтобы текст не выпадал снизу
  
  // 2. Данные самого текста
  content: RichTextParagraph;
}

```

Когда `wrap: true`, система жестко фиксирует `width` и пускает текст вниз. Если при этом включен `autofit`, система начинает сжимать текст, чтобы он не превысил еще и `height`.

Для поддержки многоколоночного текста (как в газетах, журналах или сложных интерфейсах вроде Notion/Figma) модель данных должна стать **иерархической**.

Вам нужно подняться на один уровень выше: вместо одного текстового блока создать структуру, которая управляет сеткой или списком колонок, где каждая колонка имеет свою ширину, но все они находятся внутри единого контейнера (холста).

Есть два принципиально разных подхода к реализации колонок в Rich Text. Выберите тот, который подходит под вашу задачу.

---

## Подход 1. Независимые колонки (Layout-подход)

*Каждая колонка — это отдельный независимый текстовый контейнер со своими параграфами. Текст из колонки А **не перетекает** в колонку Б.*

Этот вариант идеален для конструкторов сайтов (Tilda, Wix) или графических редакторов (Figma, Canva), где пользователь сам решает, какой текст будет в левой колонке, а какой — в правой.

### TS Модель

```typescript
// Модель одной колонки
export interface Column {
  id: string;
  width: number; // Ширина конкретной колонки (в px или процентах, например 50%)
  paragraphs: RichTextParagraph[]; // У каждой колонки свой массив параграфов
}

// Общий контейнер для многоколоночного текста
export interface MultiColumnTextBlock {
  id: string;
  x: number;
  y: number;
  width: number;  // Общая ширина всего блока на холсте
  height: number; // Общая высота
  
  gap: number;    // Расстояние между колонками (отступ)
  columns: Column[]; // Массив колонок
}

```

### Как это считает Wrap и Autofit:

* **Wrap:** Рассчитывается индивидуально для каждой колонки исходя из её собственной `width`.
* **Autofit:** Может работать независимо для каждой колонки (текст уменьшается только в той колонке, которая переполнена) или глобально для всего блока.

---

## Подход 2. Перетекающий текст (CSS Columns / Газетный подход)

*Текст представляет собой **один единый массив параграфов**, но движок рендеринга автоматически «разрезает» его и переносит остаток во вторую и третью колонки, когда первая заполняется по высоте.*

Этот вариант сложнее в реализации, но необходим для текстовых процессоров (Google Docs, Word) или читалок книг.

### TS Модель

```typescript
export interface FlowingMultiColumnBlock {
  id: string;
  x: number;
  y: number;
  width: number;       // Общая ширина холста
  height: number;      // Фиксированная высота (критично для этого подхода!)
  
  // Настройки колонок
  columnCount: number; // Количество колонок (например, 3)
  columnGap: number;   // Расстояние между колонками
  
  // Единый массив контента для всех колонок сразу
  content: RichTextParagraph[]; 
}

```

### Как это считает Wrap и Autofit в этом случае:

Чтобы алгоритм понял, где переносить текст, система делает математический расчет динамической ширины одной колонки (`columnWidth`):

$$\text{columnWidth} = \frac{\text{width} - (\text{columnCount} - 1) \times \text{columnGap}}{\text{columnCount}}$$

* **Wrap:** Алгоритм делает перенос строк (`wrap`), ориентируясь на полученную `columnWidth`.
* **Autofit:** Здесь `autofit` срабатывает, только если текст превысил `height` контейнера **и** при этом заполнил *все* доступные колонки. Система начинает пропорционально уменьшать `fontSize` для всего массива `content`, чтобы текст «втянулся» обратно в заданное количество колонок и высоту.

---

## Резюме: какой подход выбрать?

1. Если вы делаете **модульную сетку** (где в одной колонке текст, а в другой картинка, или просто разные независимые тексты) $\rightarrow$ Используйте **Подход 1** (Массив колонок, внутри каждой свой текст).
2. Если вы делаете **газетную верстку или читалку** (где длинный текст должен красиво разбиваться на колонки) $\rightarrow$ Используйте **Подход 2** (Один текст + настройки количества колонок в родителе).

Отлично, теперь картина полностью ясна. Раз мы сфокусированы на `TextBoxModel` (модели текстового контейнера на холсте) и собрали все требования воедино (Rich Text, выравнивание, Autofit, Wrap, размеры холста и поддержка колонок), давайте объединим это в **финальную, чистую и готовую к использованию TypeScript-архитектуру**.

Мы реализуем **модульный подход** (независимые колонки), так как в контексте графического холста (`TextBoxModel`) это самый гибкий и управляемый вариант.

---

### Финальная TS Модель для `TextBoxModel`

```typescript
// 1. Стили выравнивания текста внутри колонки/параграфа
export type TextAlignment = 'left' | 'center' | 'right' | 'justify';

// 2. Настройки автоподбора размера (Autofit)
export interface AutofitConfig {
  enabled: boolean;
  minFontSize?: number; // Порог, ниже которого шрифт не станет (чтобы текст оставался читаемым)
  maxFontSize?: number; // Максимально возможный базовый размер шрифта
}

// 3. Атомарный фрагмент стилизованного текста (Rich Text Run)
export interface TextRun {
  text: string;
  bold?: boolean;
  italic?: boolean;
  underline?: boolean;
  color?: string;        // HEX (#FFFFFF), RGB, и т.д.
  fontSize?: number;     // Индивидуальный размер фрагмента
  fontFamily?: string;
}

// 4. Структура параграфа
export interface RichTextParagraph {
  id: string;
  children: TextRun[];      // Массив текстовых фрагментов
  alignment: TextAlignment; // Выравнивание внутри колонки
}

// 5. Модель колонки текста
export interface TextColumn {
  id: string;
  width: number;                 // Фиксированная ширина этой колонки (в px)
  paragraphs: RichTextParagraph[]; // Параграфы, принадлежащие этой колонке
}

// 6. ГЛАВНАЯ МОДЕЛЬ: Текстовый блок на холсте
export interface CanvasTextBoxModel {
  id: string;
  
  // Геометрия холста (Критично для Wrap и Autofit)
  x: number;
  y: number;
  width: number;  // Общая ширина всего текстового бокса
  height: number; // Общая высота текстового бокса
  
  // Поведение текста при выходе за границы
  wrap: boolean;          // Разрешить перенос строк внутри колонок
  autofit: AutofitConfig; // Настройки автоподбора шрифта под размеры width/height
  
  // Структура колонок
  gap: number;            // Расстояние (отступ) между колонками в пикселях
  columns: TextColumn[];  // Массив колонок (минимум одна)
}

```

```json
{

            { "text": "Это левая колонка. ", "bold": true },

        }
      ]
    }
  ]

```

### Как эту модель теперь понимает ваш Layout Engine (движок рендеринга):

1. **Для `wrap`**: Движок берет конкретную колонку (например, `"col-1"`), смотрит на её `width` (`290`) и делает перенос строк (`wrap: true`) строго в пределах этих 290 пикселей.
2. **Для `autofit`**: Движок рендерит весь текст. Если суммарная высота получившихся колонок или ширина блока выходят за рамки общего `width` (600) или `height` (400) самого `TextBoxModel`, алгоритм пропорционально уменьшает базовый размер шрифта (`fontSize`) у всех `TextRun` до тех пор, пока весь контент идеально не впишется в прямоугольник $600 \times 400$, но не меньше, чем `minFontSize` (10).

Если мы посмотрим на профессиональные спецификации типографики и верстки (такие как CSS Text/Fonts Modules, стандарты дизайн-систем вроде Material Design 3, и внутренние структуры Figma/InDesign), то заметим, что для полноценного коммерческого редактора текста нам не хватает **нескольких критических пластов данных**.

Сейчас у нас есть «база»: буквы, цвет, жирность, колонки и автоподбор. Но профессиональная верстка «сломается», например, если длинное слово не поместится в колонку (нужны переносы), или если строки слипнутся по вертикали.

Ниже то, что **обязательно** нужно добавить в модель для профессиональной верстки текста.

---

## 1. Чего не хватает на уровне абзаца (`RichTextParagraph`)

Абзац в типографике — это не просто выравнивание, это управление вертикальным ритмом и поведением строк.

* **`lineHeight` (Интерлиньяж / Межстрочный интервал):** Без него строки текста будут налипать друг на друга. Может быть в пикселях (`number`) или множителем (например, `1.5` от размера шрифта).
* **`paragraphSpacing` (Отступ после абзаца):** В вебе и дизайне переносы строк часто делают не через `Enter` (двойной абзац), а настраивая красивый отступ снизу у параграфа.
* **`letterSpacing` (Трекинг / Межбуквенный интервал):** Расстояние между символами. Критично для заголовков (их часто разряжают) или мелкого текста в колонках (его сжимают).
* **`hyphens` (Переносы слов):** Логическое (`true/false`). Если слово не влезает в узкую колонку (`width: 290px`), оно должно либо разбиваться дефисом (газетная верстка), либо уходить целиком на новую строку, оставляя огромные дыры по бокам.

### Дополнение в код:

```typescript
export interface RichTextParagraph {
  id: string;
  children: TextRun[];
  alignment: TextAlignment;
  
  // ЧЕГО НЕ ХВАТАЛО:
  lineHeight?: number;       // Межстрочный интервал (например: 1.4)
  paragraphSpacing?: number; // Отступ после абзаца в px (например: 12)
  letterSpacing?: number;    // Трекинг в px (например: 0.5)
  hyphens?: boolean;         // Разрешить автоматические дефисы (сло-ва)
}

```

---

## 2. Чего не хватает на уровне фрагмента (`TextRun`)

В Rich Text отдельные слова могут иметь специфический визуальный статус.

* **`textTransform` (Регистр):** Возможность принудительно сделать текст капсом (`uppercase`), строчными (`lowercase`) или Капителью (`capitalize`), не меняя сами исходные буквы в базе данных.
* **`strikethrough` (Зачеркивание):** Забытый компаньон для `underline`. Нужен для цен со скидками или списков задач.
* **`baselineShift` (Смещение относительно базовой линии):** Индексы для формул или сносок: верхний индекс ($x^2$) или нижний индекс ($H_2O$). Без этого Rich Text не полноценен.

## Что говорит спецификация W3C?

Согласно спеке, вертикальный текст делится на два разных понятия:

1. **`writing-mode` (Направление блока):** Куда текут строки — горизонтально сверху вниз (как в Европе), вертикально справа налево (как в традиционном китайском) или вертикально слева направо.
2. **`text-orientation` (Поворот символов):** Когда строка идет вертикально, как повернуты сами буквы? Они могут стоять ровно друг под другом (апбайт-ориентация) или вся строка целиком может быть повернута на 90 градусов набок (часто используется для английских слов на корешках книг).

## Добавление в TypeScript модель

Поскольку поворот текста обычно применяется ко всему текстовому блоку на холсте (нельзя в одном абзаце писать буквы горизонтально, а в следующем вертикально), мы добавляем эти свойства на самый верхний уровень — в `CanvasTextBoxModel`.

```typescript
// 1. Направление потока текста (согласно W3C CSS Writing Modes)
export type WritingMode = 
  | 'horizontal-tb' // Стандартный: горизонтальные строки, идут Top-to-Bottom
  | 'vertical-rl'   // Вертикальные колонки, идут Right-to-Left (азиатский стиль)
  | 'vertical-lr';  // Вертикальные колонки, идут Left-to-Right (для интерфейсов и подписей)

// 2. Ориентация самих знаков внутри вертикальной строки
export type TextOrientation = 
  | 'mixed'    // Буквы латиницы поворачиваются набок, а иероглифы стоят прямо
  | 'upright'  // Все символы (включая латиницу) стоят ровно друг под другом
  | 'sideways';// Весь текст целиком повернут на 90 градусов (как повернутый блок)

// Обновленная модель контейнера
export interface CanvasTextBoxModel {
  id: string;
  x: number;
  y: number;
  width: number;
  height: number;
  
  // ЧЕГО НЕ ХВАТАЛО: Вертикальный текст и его поведение
  writingMode: WritingMode;
  textOrientation: TextOrientation;
  
  // Остальные параметры, которые мы утвердили ранее
  wrap: boolean;
  autofit: AutofitConfig;
  gap: number;
  columns: TextColumn[];
  verticalAlignment: 'top' | 'middle' | 'bottom';
  padding: { top: number; right: number; bottom: number; left: number; };
}

```

## Как это меняет логику Wrap и Autofit?

Введение `writingMode` кардинально меняет работу движка рендеринга, так как **ширина и высота меняются ролями**:

* **При `horizontal-tb` (Обычный текст):** Свойство `wrap` следит за ограничением по `width` (ширине). Если текст не влезает по горизонтали, строка переносится. `autofit` сжимает текст, если он начинает выходить за рамки `height` (высоты).
* **При `vertical-lr` / `vertical-rl` (Вертикальный текст):** Свойство `wrap` теперь следит за ограничением по **`height` (высоте)**! Так как строки растут сверху вниз, «перенос строки» означает, что текст уперся в нижнюю границу `height` и должен начать *новую вертикальную колонку* правее или левее. Соответственно, `autofit` начинает сжимать шрифт, если колонки по ширине выходят за рамки общего **`width`**.
Да, у этой модели есть **жирные минусы**. Когда мы попытались объединить в рамках одного JSON-объекта `CanvasTextBoxModel` профессиональную типографику (как в CSS/Figma) и многоколоночность с автоподбором (`autofit`), мы заложили несколько «архитектурных мин», которые взорвутся при первой же попытке написать код рендеринга.

Вот главные проблемы написанной модели:

---

### 1. Конфликт: `columns[].width` против глобального `autofit`

В модели у каждой колонки жестко прописана ширина: `width: number`.

* **В чем баг:** Если включен `autofit`, система должна уменьшать размер шрифта, чтобы текст влез в границы. Но если шрифт уменьшается, текст начинает занимать меньше места, и фиксированная ширина колонок в пикселях ломает пропорции холста.
* **Как правильно:** Ширину колонок нельзя хардкодить в пикселях внутри массива. Она должна считаться динамически на основе общего `width` контейнера, количества колонок и `gap` (отступа). Внутри `columns` должен быть максимум коэффициент пропорции (флекс-доли, например `1fr`, `2fr`).

### 2. «Взрыв мозга» для разработчика при `vertical-lr` (Вертикальный текст)

По спецификации W3C, когда `writingMode` переключается на вертикальный, понятия «ширина» и «высота» концептуально меняются местами.

* **В чем баг:** В нашей модели `gap` (отступ между колонками) и `columns[].width` неявно завязаны на горизонтальное разделение (слева направо). Если текст повернут вертикально, то «колонки» превращаются в «горизонтальные этажи» (строки), которые растут сверху вниз. Модель заставляет разработчика писать костыли, чтобы интерпретировать `width` колонки как её высоту в вертикальном режиме.

### 3. Избыточность данных и рассинхронизация (Контент + Лейаут)

Модель пытается хранить внутри себя и **чистый контент** (буквы, жирность), и **результат верстки** (`columns`).

* **В чем баг:** Если пользователь решит изменить ширину всего TextBox (`width` с 600px на 400px), вам придется запустить тяжелую функцию, которая пересчитает текст, расставит новые переносы (`wrap`), изменит `fontSize` (`autofit`), а затем *перезапишет* весь массив `columns` в базе данных. Это приводит к диким торможениям интерфейса.
* **Как правильно:** Модель данных должна хранить только настройки (Ширина, Высота, Количество колонок, Единый массив абзацев). А массив `columns` с разбитым по строкам текстом должен генерироваться **динамически «на лету» в памяти** (computed property) только для отрисовки на экране, но не храниться в JSON.

### 4. Проблема "Курицы и Яйца" для `TextRun.fontSize` и `autofit`

В модели внутри `TextRun` есть индивидуальный `fontSize`.

* **В чем баг:** Если слово «Скидка» имеет `fontSize: 32`, а обычный текст `fontSize: 16`. Когда срабатывает `autofit`, на сколько нужно уменьшить шрифт? До какого состояния?
* **Как правильно:** `autofit` работает предсказуемо только тогда, когда у самого TextBox задан *базовый* (родительский) размер шрифта (например, `baseFontSize: 16`), а у внутренних кусочков текста (`TextRun`) размер задается в **относительных единицах** или процентах (например, коэффициент `scale: 2.0` для заголовка). Тогда `autofit` просто уменьшает базовый шрифт, и вся конструкция сжимается пропорционально.

---

### Резюме: Как её исправить?

Чтобы модель не была «костыльной», нужно разделить её на две:

1. **Модель хранения (то, что пишем в БД):** Только геометрия баунтинг-бокса, настройки сетки (число колонок) и плоский массив текста.
2. **Модель рендеринга (то, что считает Layout Engine):** Временный объект, который получается после того, как алгоритм «прожевал» текст, применил `wrap` и `autofit` под текущие размеры экрана.
